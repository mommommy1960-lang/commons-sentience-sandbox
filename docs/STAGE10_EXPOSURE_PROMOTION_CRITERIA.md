## Overview

The **exposure quality tier** system formalises how well the pipeline understands
the sky-coverage and instrument-acceptance properties of each catalog.  It is a
five-level ladder:

```
none → synthetic_sample → partial_data_derived → analysis_candidate → analysis_grade
```

Each rung of the ladder requires the rung below to be satisfied first.  A result
with a low exposure quality tier is *not necessarily wrong* — it is simply one
that carries wider systematic uncertainty and should be interpreted more
conservatively.

The tier system is implemented in:

```
reality_audit/data_analysis/exposure_quality_tiers.py
```

---

## Why Exposure Quality Matters

The Reality Audit pipeline tests for spatial anisotropy in astrophysical event
catalogs.  An apparent anisotropy signal can arise from at least two distinct
sources:

1. **Genuine astrophysical anisotropy** in the intrinsic source distribution.
2. **Instrumental or survey-geometry bias** — e.g., non-uniform sky coverage,
   satellite orbit gaps, or instrument pointing constraints.

A well-calibrated **null model** must account for source (2).  If the exposure
correction is poor, the pipeline cannot distinguish a genuine anisotropy from an
instrumental artefact.

The exposure quality tier records *how well* the pipeline handles this
correction.  Tiers below `analysis_candidate` indicate that the null model is
approximate or proxy-only, and results should be reported with that caveat.

## The Five Exposure Quality Tiers

### Tier: none

**Constant:** `TIER_NONE = "none"`

The lowest tier.  No exposure data exists, or the catalog uses a null model
(e.g., isotropic) that explicitly does not require exposure correction.

**Typical situations:**
- IceCube HESE, which uses an isotropic null model by design.
- A catalog for which no livetime or coverage data has ever been deposited.
- A catalog whose coverage data is entirely missing from the pipeline.

**Interpretive impact:**  
Results at this tier must be treated as exploratory regardless of their
statistical strength.  Exposure correction is either inapplicable or has not
been performed.

---

### Tier: synthetic_sample

**Constant:** `TIER_SYNTHETIC = "synthetic_sample"`

A synthetic or randomly-generated exposure file has been deposited.  The data
do not represent real instrument observations and have no calibration value
beyond testing that the pipeline plumbing works.

**Promotion requirements:** None (any non-empty metadata reaches at least this
tier automatically).

**Typical situations:**
- Unit-test mock exposures generated by the test suite.
- Placeholder files written before real data are available.

**Interpretive impact:**  
Results using synthetic exposure should never be reported as scientifically
meaningful.  The tier exists to allow pipeline plumbing tests to proceed.

---

### Tier: partial_data_derived

**Constant:** `TIER_PARTIAL = "partial_data_derived"`

Exposure is derived from real observational data (i.e., the real catalog's
temporal or spatial distribution), but significant gaps or limitations remain.
This is the current operational tier for Fermi and Swift.

**Promotion requirements:**
- `non_synthetic_source = True`
- `minimum_event_count_present = True`

**Typical situations:**
- An empirical-proxy histogram built from the observed GRB sky distribution,
  used as a stand-in for the true instrument acceptance map.
- A temporal window file built by binning catalog observation dates, without
  formal livetime accounting.

**Interpretive impact:**  
Results are plausible and scientifically interesting but the null model is
known to be approximate.  Caveats should be explicitly stated in any report.
The tier does not include instrument response modelling or validated livetime.

---

### Tier: analysis_candidate

**Constant:** `TIER_CANDIDATE = "analysis_candidate"`

Real data, documented provenance, bounded and enumerated caveats.  The
exposure model is good enough for initial analysis and can support a
*first-results* note, but the instrument response model is still approximate
or absent.

**Promotion requirements (cumulative from partial_data_derived):**
- `non_synthetic_source = True`
- `minimum_event_count_present = True`
- `sufficient_observation_window = True` — multi-year or otherwise documented
  observation window.
- `explicit_provenance_documented = True` — the creation chain (what data,
  what steps) is recorded.
- `bounded_caveats_documented = True` — known limitations are enumerated, not
  left implicit.

**Interpretive impact:**  
Suitable for a candidate first-results interpretation.  Claims of anisotropy
must still be hedged: the instrument response model is approximate.  External
review is recommended before stronger claims are made.

---

### Tier: analysis_grade

**Constant:** `TIER_GRADE = "analysis_grade"`

Mission-grade exposure quality.  The instrument response function is
explicitly modelled, livetime has been cross-validated, and sky coverage
fraction has been assessed and documented.

**Promotion requirements (cumulative from analysis_candidate):**
- `non_synthetic_source = True`
- `minimum_event_count_present = True`
- `sufficient_observation_window = True`
- `explicit_provenance_documented = True`
- `bounded_caveats_documented = True`
- `instrument_response_modeled = True` — an explicit instrument response
  function (IRF) or effective-area model is incorporated.
- `livetime_validated = True` — the livetime estimate has been cross-checked
  against an independent source.
- `sky_coverage_assessed = True` — the sky coverage fraction and directional
  sensitivity map are documented.

**Interpretive impact:**  
Suitable for confirmatory scientific claims if all other pipeline criteria
(preregistration, trial correction, cross-catalog replication) are also
satisfied.  This tier is currently not reached by any catalog in this pipeline.

## Promotion Criteria in Detail

The table below summarises all eight boolean flags and which tier each enables:

| Requirement flag                  | Tier enabled (first occurrence) |
|-----------------------------------|---------------------------------|
| *(any non-empty metadata)*        | `synthetic_sample`              |
| `non_synthetic_source`            | `partial_data_derived`          |
| `minimum_event_count_present`     | `partial_data_derived`          |
| `sufficient_observation_window`   | `analysis_candidate`            |
| `explicit_provenance_documented`  | `analysis_candidate`            |
| `bounded_caveats_documented`      | `analysis_candidate`            |
| `instrument_response_modeled`     | `analysis_grade`                |
| `livetime_validated`              | `analysis_grade`                |
| `sky_coverage_assessed`           | `analysis_grade`                |

Requirements are **cumulative**: to reach `analysis_candidate` you must also
satisfy the two `partial_data_derived` requirements.

## Data-Derived ≠ Strong Input Quality

A common source of confusion is the phrase "data-derived".  In this context:

- **Data-derived** means the exposure proxy was constructed from *real
  observational data* (e.g., the GRB catalog itself, or its temporal
  distribution).
- **Data-derived ≠ analysis-grade** because:
  - The catalog's spatial distribution is influenced by the very
    instrument-acceptance non-uniformity we are trying to correct for.
    Using the data itself as the exposure model can partially obscure or
    mimic an anisotropy signal.
  - The temporal distribution of detected events is not a direct measure of
    livetime; it folds in source rate variations, trigger thresholds, and
    other non-acceptance effects.
  - No independent cross-check (e.g., a blank-sky exposure map or instrument
    simulation) is performed.

The `partial_data_derived` tier acknowledges that the proxy is better than
nothing but is explicitly *not* the same as a formal instrument exposure model.

## What Blocks Stronger Confirmatory Interpretation

The following conditions must *all* be satisfied before a confirmatory
(non-exploratory) scientific interpretation is warranted:

1. **Exposure quality ≥ analysis_candidate for all catalogs** — currently
   not reached by any catalog.

2. **Preregistration plan locked** — implemented in Stage 14.

3. **Trial correction applied** — Holm correction over primary statistics,
   implemented in Stage 14.

4. **Cross-catalog replication** — at least two independent catalogs must
   show consistent results.

5. **Systematics documented** — known limitations must be explicitly stated.

6. **No metaphysical language** — results must not claim consciousness,
   intentionality, or non-physical causation.

Currently the **primary blocker for confirmatory interpretation** is the
exposure quality tier: all three catalogs are below `analysis_candidate`.
Even if all other conditions are met, the `confirmatory_exposure_quality_readiness`
field in the Stage 14 manifest will report `not_ready` until this changes.

## How to Use the Assessment Functions

### Assessing a single catalog

```python
from reality_audit.data_analysis.exposure_quality_tiers import assess_exposure_quality

metadata = {
    "non_synthetic_source": True,
    "minimum_event_count_present": True,
    "sufficient_observation_window": False,
    "explicit_provenance_documented": True,
    "bounded_caveats_documented": True,
    "instrument_response_modeled": False,
    "livetime_validated": False,
    "sky_coverage_assessed": False,
}

result = assess_exposure_quality("fermi", metadata)
print(result["exposure_quality_tier"])
# → partial_data_derived
print(result["exposure_quality_evidence_summary"])
# → [fermi] Real observations present with minimum event count, ...
```

### Comparing tiers

```python
from reality_audit.data_analysis.exposure_quality_tiers import compare_tiers, worst_tier, best_tier

compare_tiers("none", "partial_data_derived")  # → -1
compare_tiers("analysis_candidate", "analysis_candidate")  # → 0
compare_tiers("analysis_grade", "partial_data_derived")  # → 1

worst_tier(["partial_data_derived", "analysis_candidate"])  # → "partial_data_derived"
best_tier(["none", "partial_data_derived"])  # → "partial_data_derived"
```

### Building a manifest summary (Stage 14)

```python
from reality_audit.data_analysis.mission_calibration_loader import (
    build_exposure_quality_summary_for_manifest,
)

summary = build_exposure_quality_summary_for_manifest()
print(summary["worst_exposure_quality_tier"])
print(summary["confirmatory_exposure_quality_readiness"])
# → "not_ready"
```

### Loading an exposure window file

```python
from reality_audit.data_analysis.time_exposure_loader import load_exposure_window

result = load_exposure_window("fermi", "v1")
```

## Glossary

**Exposure correction** — adjustment of the null model to account for
non-uniform sky coverage and instrument acceptance, so that an isotropic
intrinsic distribution produces a flat angular statistic.

**Empirical proxy** — an exposure model derived from the observed event
distribution rather than from a formal instrument model.  Fast to construct
but potentially biased.

**Instrument response function (IRF)** — a formal model of the instrument's
detection efficiency as a function of direction, energy, and other observables.

**Livetime** — the total time during which the instrument was operating and
sensitive to events.  Gaps (safe-mode, slew, SAA passages) reduce effective
livetime.

**SAA** — South Atlantic Anomaly, a region of enhanced particle flux that
causes instrument shutdowns for Fermi and Swift.

**Confirmatory** — a result generated according to a pre-specified, locked
analysis plan.  Distinct from exploratory analyses generated post-hoc.

**First-results note** — an internal report suitable for team review, not yet
submitted for external peer review.
